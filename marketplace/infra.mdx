<Note>
Brain Map operator feature. Configured per account, not self-serve. It does not appear in the customer marketplace; it is enabled per box out of band.
</Note>

One bundle for the operator's own monitoring stack. It combines infra metrics (PromQL queries against Prometheus, LogQL queries against Loki, and saved Grafana dashboards rendered as tiles) with infra agents (the health snapshots and helpdesk cases the monitoring agents produce: node status, container checks, recent warnings, open incidents).

## What you need

- A Prometheus URL scraping node_exporter (`PROM_URL`).
- A Loki URL for logs (`LOKI_URL`).
- A Grafana base URL plus the dashboards you want to render (uid + slug + key, optional `params`) in the tenant config `infra.dashboards`.
- The node list for the OPS_CENTER cards (`infra.nodes`): label plus PromQL selector per node.
- For the agent snapshots and cases: the infra-health files mounted into the container (synced from wherever the agents write them if the box is not the writer).

## Setup

<Steps>
  <Step title="Turn on the feature flag">
    Set `features.infra: true` in that box's tenant config (e.g. `tenant.demo.json`) or flip it in the settings table.
  </Step>
  <Step title="Point it at the stack">
    Set `PROM_URL`, `LOKI_URL`, the Grafana base (`infra.grafanaBase`), and the saved dashboards list (`infra.dashboards`) via env or tenant config.
  </Step>
  <Step title="Configure the node cards">
    Set `infra.nodes` as a list of `{ key, label, selector }` where `selector` is a brace-less PromQL label set, for example `instance="controlnode",job="integrations/unix"`. The cards run CPU/MEM/DISK/LOAD against each selector. An empty list falls back to neutral placeholders.
  </Step>
  <Step title="Set dashboard params">
    Each `infra.dashboards` entry may carry a `params` string of Grafana template vars appended to the embed URL, for example `var-instance=10.20.10.201`. The embed drops full kiosk mode so the variable dropdowns stay usable.
  </Step>
  <Step title="Mount the infra-health directory">
    For the agents side, mount the infra-health directory (the agents' `latest.md`, hourly snapshots, and `helpdesk/cases/`) at `/infra-health` in the api container.
  </Step>
</Steps>

For cross-site Grafana embedding (app and Grafana on different domains), Grafana needs `allow_embedding=true` plus either anonymous Viewer access or a same-site session, and the app's CSP must permit it (`frame-src 'self' https:` in `apps/web/nginx.conf`). A missing `frame-src` makes the browser block the frame.

## How it works

- `query_metric` runs an instant PromQL query. Only node_exporter host metrics are scraped (CPU, memory, disk, load, network, up). There are no container or application metrics, so the agent says so rather than invent one.
- `query_logs` runs a LogQL selector against Loki (labels: host, container, namespace, service_name, job), newest first.
- `show_dashboard` renders a saved Grafana dashboard as an embedded tile.
- `infra_health` reads the latest agent snapshot, `infra_cases` lists recent helpdesk cases, and `read_infra_file` opens a specific snapshot or case file by name.
- Everything here is read-only.

## What you can ask

- "what is CPU on node-1" / "memory free on node-2"
- "any errors in the app logs in the last hour"
- "show the node overview dashboard"
- "how is the infra" / "what do the agents report"
- "any open cases" / "recent incidents"

## Limitations

- node_exporter metrics only. No container or application metrics; asking for one that does not exist gets a plain "not available", not a guess.
- Dashboards must be in the tenant config (uid + slug + key). An unknown key returns the list of valid ones.
- The agents side needs the mount. Without the infra-health directory mounted, the health and cases tools have nothing to read.
- Hidden and not self-serve. Enabled per box by the operator, not from the customer marketplace.
- The Infra page is custom per customer. The metric and log tools are generic; the page is not.
