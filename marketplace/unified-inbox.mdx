# Unified Inbox

Two things in one place. (1) A carded stream of inbound leads and messages so inquiries stop
scattering: Telegram inbound, Twenty CRM newest lead, and email/form/SMS items you push from n8n or
Zapier. (2) **Talk to your canvas agent from Telegram** (new): allowlisted chats message your bot and
the agent answers in chat, so you can drive it without logging into the platform. Replies and write
actions stay approval-gated.

## HOW TO SETUP

### Inbox capture
Marketplace -> Unified Inbox. For Telegram, set `TELEGRAM_BOT_TOKEN`. CRM (Twenty) newest-lead
fold-in is optional. Email/form/SMS arrive when n8n or Zapier push to `add_inbox_item`.

### Talk to the agent over Telegram (gateway)
1. Set `TELEGRAM_BOT_TOKEN` (from @BotFather) on the card.
2. Find your **chat id**: message the bot once, then use the chat-id helper (do this before any
   webhook is set, since the helper and a webhook cannot both be active).
3. Set **`TELEGRAM_ALLOWED_CHAT_IDS`** (comma list) to the chat ids allowed to drive the agent. This
   is the opt-in and the security boundary. Leave it blank for inbox-only (no agent).
4. The agent gateway starts automatically (at boot, or within ~20s of setting the allowlist) and
   long-polls Telegram. Message the bot and it replies.

## WHAT USER NEEDS TO GET
A Telegram bot token (from @BotFather). Your chat id for the allowlist. Twenty CRM is optional.

## HOW IT WORKS
- **Agent chat uses an outbound long-poll gateway** (`gateways/telegram.js`), the standard
  channel-adapter pattern. The box calls Telegram (`getUpdates`); there is **no public endpoint and
  no login or proxy change needed**. The webhook path (`/api/webhooks/telegram`) still exists for inbox
  capture but is dormant in poll mode (no webhook is registered).
- An inbound message from an **allowlisted** chat is routed into the canvas agent with its own
  per-chat conversation history (`telegram:<chatId>`), and the agent's text reply is sent back. A
  non-allowlisted chat is ignored. There is a small per-chat rate limit.
- **Writes stay gated.** If the agent proposes a write (send email, create lead, run a Zapier or
  Composio write), it is queued to the app's approvals queue and the chat reply says how many were
  queued; you approve in the app. Reads and answers run live.
- Inbox tools: `list_inbox` shows stored items plus your newest CRM lead; `reply_inbox` queues an
  approval-gated reply (Telegram-sourced items reply via the bot); `add_inbox_item` lets n8n/Zapier
  push items in.

## WHAT CAN BE CALLED VIA AI
From Telegram (allowlisted chat):
- "what's my pipeline?" -> agent answers in chat (read)
- "and the newest lead?" -> remembers the conversation (per-chat session)
- "draft an email to `<addr>` about X" -> queued for approval, you approve in the app

From the canvas, the inbox tools:
- "what's in my inbox?" / "any new inquiries?" -> list_inbox
- "reply to that Telegram message: ..." -> reply_inbox (approval-gated)

## LIMITATIONS
- **Allowlist is required for agent chat.** Empty `TELEGRAM_ALLOWED_CHAT_IDS` = inbox-only. Only
  allowlist chats you would hand the agent to (they get the operator's agent powers).
- **Text only in Telegram.** Canvas tiles and visuals do not render there; the agent's written
  answer is what you get.
- **Writes are approval-gated** and approved in the app for now (the grant-once / trust-rules
  direction will let approved action types run from chat later).
- **One bot per box**, single-process poll loop (these boxes are single-instance).
- Old messages sent before the gateway was running are skipped (it starts from the latest update).

## FAQ
Q: Can I talk to my agent without logging into the web app? A: Yes. Set your chat id in
`TELEGRAM_ALLOWED_CHAT_IDS`, message the bot, and the agent replies in chat.

Q: Is this secure? A: It is outbound long-poll (no exposed endpoint), gated by the chat-id
allowlist, and writes never auto-run. It does not touch the platform's sign-in.

Q: Why long-poll instead of a webhook? A: A webhook would need a login and reverse-proxy exception to be
reachable. Long-poll avoids exposing anything, so it is the more secure default for self-hosted setups.
