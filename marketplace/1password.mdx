Look up items in your 1Password vault from the MindMap chat, and reveal a single field value
behind an approval. The search side only ever shows item titles and the field LABELS (what
exists), never the values. To see an actual value, the agent queues an approval; you approve,
and the value is shown once on the tile for copying and is never written to the audit log.

## HOW TO SETUP

1. In 1Password, create a **service account** (Developer → Service Accounts) and grant it read
   access to the one vault you want MindMap to use. Copy its **token**.
2. Open MindMap → **Admin** (marketplace) → the **1Password** card.
3. Paste the **service-account token** into its write-only box and **Save**.
4. Paste the **vault name** into the Vault box and **Save** (the token is scoped to one vault,
   and item reads require naming it).
5. Flip **Enable** on.

It applies on your next message, no restart.

## WHAT USER NEEDS TO GET

- **A 1Password service-account token** (`OP_SERVICE_ACCOUNT_TOKEN`), scoped to one vault, read
  access. Treat it like any vault credential.
- **The vault name** (`OP_VAULT`) the token can read.

The container runs the official `op` CLI; the token and vault are injected into it per call from
the marketplace key store.

## HOW IT WORKS

- `op_lookup` lists items whose title matches your search and, for the top matches, the field
  **labels** they carry (password, username, card number, etc). No values.
- `reveal_secret` **queues** an approval for one field of one item. Nothing is revealed yet.
- On approve, the value is fetched and shown **once** on the tile for copying. The audit log
  records that a reveal happened (item + field + "revealed"), never the value itself.
- Common phrasings map to real field labels (e.g. "cvv" -> verification number, "exp" -> expiry
  date), so you can ask naturally.
- Token + vault are read **live** via the marketplace key store (saved to your pod, never shown
  back, never in the repo). Editing them applies on the next message.

## WHAT CAN BE CALLED VIA AI

- "find the Amazon login in 1password": lists the item + its field labels
- "what's in 1password for the company card": item + labels, no values
- "reveal the password for the router admin": queues an approval; on approve, shown once
- "show the cvv on the company card": same, approval-gated

The agent will never print or guess a secret value, and never claims a reveal happened before
you approve it.

## LIMITATIONS

- **One vault.** The service-account token is scoped to a single vault; lookups and reveals are
  within it.
- **Reveal is one field at a time**, behind an approval, shown once. There is no bulk export.
- **Labels, not values, on search.** `op_lookup` deliberately never returns values.
- **Needs the `op` CLI in the image** (it ships in the api container).

## FAQ

**Can the agent dump my whole vault?** No. Search returns titles and field labels only. Values
come one field at a time, each behind its own approval, and are shown once.

**Is the revealed value logged?** No. The audit log records that item X field Y was revealed,
never the value.

**Is my token safe?** The key box is write-only. The token is saved to your pod, never shown
back or returned to the browser, and never in the repo.

**Why do I need the vault name?** A service-account token is scoped to one vault, and 1Password
requires naming it on item reads. Without it, lookups come back empty.

**Can I use my whole 1Password account?** Use a service account scoped to just the vault you
want MindMap to touch. Do not give it your personal account credentials.
