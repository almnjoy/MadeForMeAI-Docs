Cisco Meraki network operations, run from the MindMap chat. Ask about a client in plain
language and the agent finds it across your org, pulls its status, usage, and events, checks
the WAN uplinks and the switch port or access point it depends on, and tells you the likely
problem. It can also block or unblock a client, but only after you approve. Everything is
read-only except block/unblock, and those never fire on their own.

## HOW TO SETUP

1. Open MindMap, go to **Admin** (the tools marketplace).
2. Click the **Meraki** card.
3. In the detail panel, paste your **Meraki API key** and your **Organization ID** into the
   write-only boxes and hit **Save** on each. The fields show "set" once saved and never show
   the value back.
4. Flip the **Enable** toggle on.

It applies on your next message. No restart. To turn it off, toggle it off in the same panel.

## WHAT USER NEEDS TO GET

- **A Meraki Dashboard API key.** In the Meraki Dashboard: your profile (top right) → **My
  profile** → **API access** → **Generate API key**. You need org-admin rights. Read-only is
  enough for the lookups; to use block/unblock the key's account needs write access to the
  network.
- **Your Organization ID.** The long number for your org. Quickest way: with the key set, ask
  the agent "list my meraki networks" once it's enabled, or pull it from
  `GET /organizations` in the Meraki API, or read it out of a dashboard URL.

That is it. No agent, no on-prem connector. The calls go straight to Meraki's cloud API.

## HOW IT WORKS

- MindMap calls the **Meraki Dashboard API** (`api.meraki.com`) server-side using your key.
  Your key stays in the pod, never in the browser, never in the repo.
- **find_client** is the entry point. It lists the clients on each network in your org and
  matches the one you named by IP, MAC, or hostname. It hands back a `networkId` + `clientId`
  that the other tools use.
- The other read tools take that client or network and pull one specific thing (details,
  usage, events, uplink health, switch port, wireless health, DNS/VLAN).
- **run_client_health_check** is the one-shot version: it chains the relevant reads and gives
  the agent a full bundle to summarize the likely cause and next step.
- Every result drops as a **tile** on your canvas, plus the agent's written summary.
- **block_client / unblock_client are writes.** They do NOT change anything. They queue an
  **approval card**. Only when you click approve does MindMap set the client's Meraki
  device-policy (Blocked or Normal). Reversible.

## WHAT CAN BE CALLED VIA AI

Ask in plain language. The agent picks the right tool.

- "find the client at 10.x.x.x" / by MAC or hostname: locate a client
- "what are the details on that client": status, VLAN, IP/MAC, SSID/AP or switch+port
- "how much has it been using": recent usage
- "what events has it had": connect/auth/DHCP/DNS/roam events
- "is the internet up / show uplink status": WAN uplinks, active/failed, public IP
- "any packet loss or latency on the WAN": uplink loss + latency
- "show the ports on switch <serial>": switch port status, link, errors, clients
- "how's the wireless": connection success/fail by stage (assoc/auth/DHCP/DNS)
- "what's the DNS/VLAN setup": appliance VLANs, subnets, DNS servers, DHCP
- "what's been happening on the network": recent event log
- "google isn't loading for <client>, what's wrong": full triage in one go
- "block <client>" / "unblock <client>": queues an approval, then sets the device-policy

## FAQ

**Does it change anything without asking?** No. Reads are instant. The only writes are block
and unblock, and both wait for your approval click. Nothing fires on its own.

**It says it can't find my client.** find_client only sees clients seen in the last 24 hours.
If the device is off or hasn't been on the network recently, it won't show. Try the MAC or
hostname instead of the IP.

**Is "block" a firewall rule?** No. It sets the client's Meraki **device-policy** to Blocked
(reversible per-client), which is cleaner than editing firewall rules. Unblock sets it back to
Normal.

**Is my API key safe?** The key box is write-only. It's saved to your pod and never shown back
or returned to the browser, and it's never in the repo.

**A tile came back with an error.** A few of the deeper read endpoints (client events,
switch-port shape, wireless stats) were built against the API docs and may need a small tweak
for your exact gear. Tell us which tool errored and we'll adjust it.

**Rate limits?** The Meraki API allows about 5 requests per second per org. The plugin keeps
within that; a very broad question that fans out across many networks may be a little slower.
